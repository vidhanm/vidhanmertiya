I've been following Jhey Tompkins (@jh3yy) for a while.

If you don't know him: he's a creative developer who posts some of the most genuinely useful CSS and UI work on the internet. Not theory. Not blog posts about blog posts. Actual working code for scroll-driven animations, themed custom scrollbars, micro-interactions, SVG tricks — the kind of stuff that makes you stop mid-scroll, hit "bookmark," and think I'll definitely come back to this.

You don't come back to it. Nobody does.

That's the problem.

Twitter is a great place to share things and a terrible place to find them later

Jhey has been posting on Twitter for years. Hundreds of CSS demos. Dozens of animation techniques. Code blocks that would've taken me hours to figure out myself, just... dropped into a timeline and gone.

Twitter's search is bad. The algorithm buries old posts. Even if you've been following someone for two years, good luck finding that specific scroll-animation thread from eight months ago. And Jhey doesn't have a blog. He doesn't have a docs site. His knowledge lives entirely in his timeline.

This always bothered me in a low-grade way. And then one day I just thought: someone should fix this. Then I thought: I could fix this.

So I did.

What I built

jhey-tweets.pages.dev is a searchable, categorized showcase of Jhey's best CSS and frontend tips. Think of it like a personal knowledge base built on top of his Twitter history — one place, search that actually works, organized by topic.

The stack is simple: vanilla JavaScript frontend, a compiled JSON database, hosted on Cloudflare Pages. The whole thing is open source at github.com/vidhanm/jhey-tweets.

But the interesting part wasn't the frontend. It was getting the data into shape.

5,698 tweets is not 5,698 tips

The scraper (built on tweet-harvest) pulled Jhey's entire timeline — 5,698 tweets total. But raw tweet data is noisy in ways that aren't obvious until you're staring at it.

A lot of Jhey's posts are replies. He's active in conversations, answering questions, riffing with other developers. That's genuinely useful content but it's not the same as a self-contained CSS demo. If I included every tweet, the database would be full of half-sentences ripped from threads, responses without context, and "haha yeah exactly" messages that mean nothing in isolation.

So I wrote a compiler (organize_data.js) that does a few things:

The 6 categories everything maps to

✨ Scroll Animations 🧑‍🍳 Custom Scrollbars 🖱️ Micro-interactions 🎨 SVGs & Canvas ⚡ CSS Hacks & Layouts 💭 Insights

Thread grouping. Jhey often posts multi-tweet threads — a demo in the first tweet, the code in the second, a refinement note in the third. The compiler groups these by conversation ID and chains them chronologically, so you get the whole thought, not a fragment.

Chat filtering. If Jhey is in a conversation with someone else, the thread gets filtered out unless it contains a CSS or JS code block, or has attached media. This preserves his actual code answers to questions (which are often incredibly useful) while dropping the standard conversational noise.

Automatic categorization. Every surviving thread gets classified into one of six topics. This isn't perfect — keyword matching never is — but it's good enough that browsing by category actually feels useful.

After all that processing: 5,698 raw tweets became 1,701 curated tips. That's not a 70% loss. That's a 70% signal improvement.

The thing that surprised me

I expected the output to be mostly "look at this cool demo" posts. Eye candy. Impressive but not always instructive.

That's not what the data shows.

A huge chunk of Jhey's posts are genuinely explanatory. He doesn't just ship a working animation — he explains why it works, what the CSS property is doing, why the old approach was worse. The code blocks are almost always accompanied by a plain-English sentence about the underlying mechanic.

That changes what this database is. It's not a gallery. It's closer to a curriculum.

If someone wanted to learn scroll-driven animations in CSS from scratch, they could probably do it entirely from this database. Same for custom scrollbar styling, same for a lot of the micro-interaction patterns. The knowledge is all there. It was just scattered.

Why I didn't use a framework

The whole frontend is vanilla JavaScript. No React, no build step, no bundler.

This was deliberate. The database is a single JavaScript file (jh3yy_database.js) with 1,701 entries. Loading all of them on mount would be slow. But with plain JS, I could control exactly what renders and when: six cards initially, more on "Load More," iframes only mounting when they come into view.

The result is a page that loads instantly even though it's technically serving a massive dataset. I also wanted anyone who clones the repo to be able to understand it in ten minutes. No configuration, no npm install, no environment variables. Open index.html in a browser and it works.

The scraper is the real product

The showcase is useful now. But the scraper is what makes it durable.

It runs incrementally. Every time you run it in --mode=new, it finds the newest tweet in the local CSV, uses that as the since_id, and fetches only what's been posted since. Every time you run it in --mode=old, it goes backwards from the oldest tweet, filling in historical gaps.

This means the database stays current without re-scraping everything from scratch. Jhey posts new stuff every week. The scraper can keep up.

I could have built a static snapshot and called it done. Instead the architecture treats the database as a living thing — not an archive of what Jhey used to post, but a directory of what he keeps posting.

What this is actually about

I think a lot about the gap between knowledge being produced and knowledge being findable.

Jhey is genuinely one of the best CSS educators on the internet. But he doesn't call himself an educator. He just posts what he's building, what he's learned, what he thinks is interesting. The knowledge is real and it's high-quality and most of it just... disappears into the timeline.

This project is a small fix for that. A way of saying: this stuff is worth keeping.

There's probably a version of this for every prolific technical creator on Twitter — people who've been shipping knowledge for years with no structured way to reference it. That feels like a real problem worth solving at larger scale.

For now, I solved it for Jhey's corner of the internet. The database is live, it's searchable, and it's open source. If you use it and find something worth bookmarking, that's kind of the point.
