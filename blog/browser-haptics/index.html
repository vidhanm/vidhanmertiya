Apple made a movie trailer vibrate in sync with an F1 car. I tried to do the same thing in a browser.

The Web Vibration API exists. Most developers don't know about it. Here's what happens when you actually push it, and where it hits a wall called iPhone.

I'm a movie person. I like F1 too: the sport, the sound, the whole thing. So when the trailer for Apple's F1 film dropped, I watched it.

But I didn't watch it on a browser. I watched it in the Apple TV app on an iPhone. And while the car was accelerating, tires screaming, and gearbox slamming through shifts, my phone was vibrating. Not randomly. In sync. The haptic feedback matched the engine note, the braking, and the acceleration, beat for beat.

It felt like holding a tiny game controller tuned to a Formula 1 car.

I sat there for a second and thought: can a browser do this?

It turns out the web has an API for this. Nobody uses it.

The Web Vibration API has existed since 2012. One function: navigator.vibrate(). You pass it a duration in milliseconds, or an array alternating between vibration and pause durations, and the device buzzes.

That's it. The whole API.

It's supported on Android in Chrome, Firefox, and most Chromium-based browsers. It works. You can call navigator.vibrate([200, 100, 400]) and feel it.

The reason almost nobody builds with it is the same reason I found out about it the hard way: it doesn't work on iPhone. Apple has never implemented the Vibration API in Safari. Zero support. When you call navigator.vibrate() on an iPhone, nothing happens. Not an error, not a warning; just silence.

For most developers, that's a dead end. If it doesn't work on half the phones in the world, why build with it?

I built with it anyway.

The build: syncing vibrations to a video timeline

The core idea is simple: load a video, define a list of haptic events as timestamps, and as the video plays, fire the right vibration at the right moment.

Every haptic event is an object with three fields: the timestamp it fires at, how long it lasts, and how intense it should feel. The video's timeupdate event fires roughly every 250ms, so you get a check four times a second, which is good enough resolution for a motor racing haptic track.

1.2s: Accel burst
2.6s: Hard brake
3.4-4.0s: Gear shifts
4.7s: Straight burst

7.8-8.8s: Rev match
0:000:020:040:060:080:10
Engine / Acceleration Brake / Impact

Between the named events, there's a background RPM pattern: a low-intensity buzz that pulses like an engine idling. It's not glamorous but it fills the silence between events in a way that feels right. You stop noticing it consciously, but if you mute it, something feels missing.

The intensity slider scales all vibration durations proportionally. Turn it to 50% and everything buzzes for half as long. It's a simple proxy for 'intensity,' since the Vibration API gives you no control over the strength of the vibration, only its duration.

The platform split

Here's where the project gets honest about its limits.

Platform API Used Status Notes
Android (Chrome) navigator.vibrate() Works Full sync, intensity scaling, background RPM pattern all functional
iOS 18+ (Safari) input[switch] hack Limited System haptic tick on each event, no duration/intensity control
iOS < 18 N/A No haptic support No vibration API support; falls back to on-screen event flash
Desktop N/A No haptic support No vibration hardware; demo still runs, events show on screen

Android is where this actually works as intended. The Vibration API does exactly what you'd hope: you fire an event, the phone buzzes with the right timing, and the sync is close enough to feel intentional.

iOS is the interesting story.

The iOS 18 trick nobody talks about

Apple has never shipped navigator.vibrate() in Safari. It's been in the spec for over a decade. They just… chose not to implement it. Their reasoning is presumably privacy and UX control. A webpage shouldn't be able to make your phone buzz without you knowing. But whatever the reason, the door has been closed for years.

Then in iOS 18, something interesting shipped quietly: a new HTML input type called switch. It renders a native iOS toggle control. When you programmatically toggle it, clicking it via JavaScript, WebKit fires a system haptic tick. The same tap feedback you feel when you flip a toggle in Settings.

It's not designed as a haptics API. It's a UI component that happens to have a side effect. But if you toggle it fast enough, at the right moments, you get something resembling rhythmic haptic feedback.

The iOS 18 Haptic Trick: How It Works
// Hidden in the page: zero size, off-screen <input type="checkbox" switch id="ios-haptic" style="opacity:0; position:absolute; pointer-events:none;" /> // When a haptic event fires: function triggerIOSHaptic() { const el = document.getElementById('ios-haptic'); el.checked = !el.checked; // toggle = 1 haptic tick // For a "long" vibration, chain multiple toggles setTimeout(() => { el.checked = !el.checked; }, 80); setTimeout(() => { el.checked = !el.checked; }, 160); }
⚠️ This only works in Safari on iOS 18+. Other iOS browsers (Chrome, Firefox on iOS) are WebKit wrappers and may not expose the same haptic behavior. Duration and intensity are not controllable; you get one tick per toggle.

It's a hack. A beautiful, practical hack. The haptic you get is a single system tick, not a sustained buzz. You can chain multiple toggles to approximate a longer event, but you're stuck with iOS's default haptic strength: no intensity scaling.

It's a far cry from what Apple did with that F1 trailer. But it's something. On iOS 18+, opening this in Safari, you'll feel a tick on each engine event. The sync is there. The texture isn't.

The audio analysis feature I didn't fully finish

The site has a "Generate Vibrations" button. The idea: upload any video, analyze the audio waveform, and auto-generate the haptic timeline without manually defining every timestamp.

The approach uses the Web Audio API's AnalyserNode. We decode the audio, walk through it in 50ms windows, measure the average amplitude, and any window above a threshold becomes a haptic event. The duration of the vibration scales with how loud that moment is.

Auto-detected haptic events from audio amplitude analysis
Each bar = 50ms audio window. Bar height = amplitude. Red bars cross the haptic threshold and vibration fires.

It works for clean, percussive audio: a car engine revving up has enough amplitude variation that the events are picked out accurately. For more complex audio it produces noise. You'd need frequency analysis and maybe onset detection to do it properly.

What I shipped is a working proof of concept that's genuinely useful for certain types of content. What it would need to be great is a proper signal processing pass, which is a different project.

What Apple actually did, and what the gap looks like

The F1 trailer haptics on iPhone weren't built with navigator.vibrate(). Apple controls the entire stack: the OS, the haptic engine, the media player, and the CoreHaptics API. They could design an AHAP (Apple Haptic and Audio Pattern) file alongside the video, sync it frame-perfectly with the Core Haptics engine, and ship it as part of the trailer experience.

Core Haptics gives you real waveform control: frequency, intensity, sharpness, attack, decay. You can actually simulate the texture of a tire on tarmac, not just a generic buzz.

The web gives you a light switch. Apple gave that F1 trailer a mixing board. The interesting question isn't "why can't the web do this yet"; it's "what would a real browser haptics standard actually look like?"

There's a W3C proposal called the Gamepad Haptics API that goes further, and gamepad haptics already have intensity and waveform control in Chrome. The same concepts could extend to mobile haptics. It just hasn't happened yet.

Why I built it anyway

The honest reason: I thought it would be fun, and I wanted to know how far the web could get.

The answer is: on Android, surprisingly far. On iPhone, further than zero but nowhere near enough. On desktop, it runs but you're just watching the events tick by on screen like a visualizer.

It's a project that works best if you open it on a mid-range Android phone, hit play, and just sit with it. The sync between the video and the buzz in your hand is real. It's not Apple's trailer. But it's not nothing either.

The code is a single index.html with no build step, no framework, and no dependencies. Open it in a browser. It works or it doesn't, depending on your phone.

That feels like the right format for an experiment about something the web can almost do.
